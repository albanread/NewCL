# NCL Windows FFI

*Status: implemented through the current metadata-backed Win32 surface. Last updated 2026-05-23.*

## What this is

This document describes the Windows FFI that exists in NCL today.
The previous version of this file was mostly a forward-looking plan.
That is no longer accurate: the runtime already has a working Windows
surface, a raw `%ffi-call` entry point, metadata-backed `win32` /
`defwin32` helpers, foreign-buffer support, and callback trampolines.

The right way to read this file now is:

- what is implemented
- what the current Lisp surface looks like
- what is still deferred

## Runtime shape

When NCL starts with `--windows`:

- thread 0 becomes the Win32 UI thread
- thread 0 creates a hidden `HWND_MESSAGE` dispatcher window
- thread 0 runs the Win32 message pump
- a worker thread named `ncl-lisp-worker` runs the Lisp session
- the driver tries to load `packs/windows_api.pack`

Without `--windows`:

- Lisp runs directly on thread 0
- the Windows surface is disabled
- `(windows-enabled-p)` returns `NIL`

This split is implemented in `src/ncl-driver/src/main.rs` and
`src/ncl-runtime/src/win_surface.rs`.

## What exists today

### 1. Windows surface predicates and UI dispatch

These native functions are always installed:

- `(windows-enabled-p)`
- `(ui-thread-id)`
- `(ui-thread-p)`
- `(%ui-execute closure)`

`Lisp/Library/win32-threading.lisp` builds the public macros on top:

- `(on-ui-thread body...)`
- `(post-to-ui-thread body...)`

Important current behavior:

- `on-ui-thread` is real and synchronous
- nested `on-ui-thread` calls short-circuit on the UI thread
- `post-to-ui-thread` is still a temporary alias to the synchronous
  path; it does not yet use a true `PostMessage` fire-and-forget carrier

So the UI-thread story is implemented, but only the blocking RPC path is
complete.

### 2. Raw `%ffi-call`

The runtime exports:

```lisp
(%ffi-call DLL FN ARG-TYPES RETURN-TYPE ARG1 ARG2 ...)
```

This lives in `src/ncl-runtime/src/win_ffi.rs` and is implemented on
Windows x64 today.

What it does:

- caches `LoadLibraryW` results per DLL
- caches `GetProcAddress` results per `(DLL, function)` pair
- marshals Lisp values to native integer/pointer arguments
- calls the target function through an x64 dispatcher
- marshals the return value back to a Lisp `Word`

Supported argument/return kinds today:

- `:void`
- `:i8` `:u8`
- `:i16` `:u16`
- `:i32` `:u32`
- `:i64` `:u64`
- `:isize` `:usize`
- `:bool`
- `:handle`
- `:ptr`
- `:wstr`
- `:cstr`

Current limits:

- maximum arity is 12
- float arguments and returns are not implemented yet
- bignum integer marshalling is not implemented yet
- the raw `%ffi-call` shim does not yet honor a `:route` keyword;
  every call runs on the calling thread

That last point matters: UI-thread routing exists at the Lisp surface,
but not yet inside the raw variadic FFI kernel itself.

### 3. Metadata-backed Win32 lookup and dispatch

At `--windows` startup the driver tries to load a binary metadata pack:

- `packs/windows_api.pack`

The loader lives in `src/ncl-runtime/src/win_metadata.rs`.
The pack is generated by:

- `scripts/generate_win32_pack.py`

from the external SQLite database derived from WinMD metadata.

Once loaded, the runtime exposes:

- `(%win32-lookup name)`
- `(%win32-call name &rest user-args)`

`%win32-lookup` returns metadata such as:

- DLL name
- argument tags
- return tag
- `set_last_error` flag
- route policy (`:ui` or `:any`)

`%win32-call` uses that metadata to perform a dynamic one-shot call.

### 4. Lisp surface: `win32` and `defwin32`

`Lisp/Library/win32.lisp` provides two user-facing entry points.

Cold path:

```lisp
(win32 "MessageBoxW" hwnd text caption flags)
```

Hot path:

```lisp
(defwin32 message-box-w "MessageBoxW")
```

Behavior:

- `win32` does metadata lookup on each call and then dispatches
- `defwin32` performs metadata lookup at macroexpansion time and bakes
  a regular `defun`
- if the metadata route is `:ui`, `defwin32` wraps the call in
  `on-ui-thread`

So route-aware UI dispatch exists today at the metadata/Lisp layer even
though raw `%ffi-call` still runs directly.

## Foreign buffers

NCL already has native foreign-buffer primitives installed by the
compiler:

- `MAKE-FOREIGN-BUFFER`
- `FREE-FOREIGN-BUFFER`
- `BUFFER-ZERO`
- `BUFFER-REF-*`
- `BUFFER-SET-*`
- `BUFFER-READ-WSTRING`
- `BUFFER-WRITE-WSTRING`

`Lisp/Library/win32-buffer.lisp` builds `defstruct-win32` on top of
those primitives. That gives Lisp code a practical way to work with
Win32 structs such as `RECT`, `POINT`, `SIZE`, and `MSG`.

This is not speculative design work anymore; it is part of the current
surface.

## Callbacks

Win32 callback support is also present.

Runtime side:

- `src/ncl-runtime/src/win_callback.rs`

Compiler/JIT hook:

- `%MAKE-WIN32-CALLBACK` is installed from `ncl_llvm`

Lisp surface:

- `Lisp/Library/win32-callback.lisp`
- `(make-win32-callback (params...) body...)`
- `(define-win32-callback name (params...) body...)`

The model is:

- register a Lisp closure in a process-global callback table
- JIT-emit a native trampoline that Win32 can call
- have the trampoline route back through `ncl_callback_dispatch`
- use the UI thread mutator when the OS calls the callback there

The trampoline currently lives for the process lifetime.

## What the old doc got wrong

The previous version of this document is behind the code in several
important ways.

It said or implied that these were still future work:

- the `--windows` driver split
- the hidden UI dispatch window
- `on-ui-thread`
- raw `%ffi-call`
- metadata-backed `win32` / `defwin32`
- foreign buffers
- callback trampolines

Those pieces now exist.

## What is still pending

Several pieces are still intentionally incomplete.

### Raw FFI routing

`%ffi-call` does not yet honor a route keyword. The comment in
`win_ffi.rs` is explicit: every raw call still behaves like `:any`.
UI-thread routing is handled one level up today.

### Float ABI support

The current dispatcher handles integer/pointer-register arguments.
`f32` / `f64` support still needs the XMM-register path.

### True fire-and-forget UI posting

`post-to-ui-thread` is still a synchronous alias, not a real queued
`PostMessage` implementation.

### Richer Win32 type coverage

The metadata pack and pack generator still skip types that are awkward
for the current ABI layer:

- callbacks encoded directly in metadata signatures
- many by-value structs
- COM interfaces
- complex unions

Those still need either richer type support or manual wrappers.

### Error-surface polishing

The metadata layer carries `set_last_error`, but the old document's
planned user-facing `(win32-last-error)` story is not the main current
interface surface. The shipped code is centered on direct return-value
marshalling plus metadata-driven lookup and route selection.

## Practical guidance

Use the surface in this order:

1. For ad hoc calls, use `(win32 "FunctionName" ...)`.
2. For stable bindings, use `(defwin32 ...)` and call the generated
   function.
3. For struct traffic, use `defstruct-win32` and foreign buffers.
4. For callback-based APIs, use `define-win32-callback`.
5. Use `on-ui-thread` when the API must run on the UI thread.

Drop to raw `%ffi-call` only when you need a call that the metadata pack
or helper macros do not cover cleanly.
